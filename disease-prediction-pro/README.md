# Disease Prediction System

An ML-based symptom checker: select symptoms → get the most likely disease, its
description, precautions, suggested medications, and diet recommendations.

**Educational demo only — not a medical device and not a substitute for professional
diagnosis.**

## Dataset

Public "Disease Prediction Using Machine Learning" dataset (derived from the Columbia
University DiseaseSymptomKB, widely used for this exact task):
- **4,920 patient records**
- **132 binary symptom features**
- **41 disease classes** (balanced, 120 records each)
- Supplementary tables: disease descriptions, precautions, medications, diets, and
  symptom severity weights

## How it was built (like a real ML pipeline, not a toy)

1. **Data cleaning** — stripped whitespace artifacts in column/label names present in
   the raw CSV.
2. **Stratified train/test split** (80/20) so every disease is represented in both sets.
3. **Model comparison with 5-fold stratified cross-validation** across six standard
   classifiers: RandomForest, GradientBoosting, SVM (RBF), Logistic Regression,
   GaussianNB, and KNN — not just picking one algorithm and hoping for the best.
4. **Best model selected by CV score**, refit on the full training set, then evaluated
   on the untouched held-out test set (accuracy, per-class precision/recall/F1,
   confusion matrix — all saved to `model/metrics.json`).
5. **Final model refit on all data** before saving, so the deployed model uses every
   available record.
6. **Recommendation layer** — the predicted disease is joined against the description,
   precautions, medications, and diet tables so the app returns actionable info, not
   just a label.

### An honest caveat
Cross-validation and test accuracy both come out at ~100%. That's a property of this
dataset — each disease maps to one fixed, non-overlapping symptom combination
(repeated 120 times), so it's a clean, well-separated classification problem. It's a
good, realistic dataset for *learning the ML pipeline end-to-end*, but real clinical
symptom presentations overlap heavily between conditions. Don't read the 100% as "this
diagnoses real patients" — it demonstrates the pipeline is correct, not that it's
medically validated. The README caveat is also shown inside the running app.

## Project structure
```
disease-prediction-pro/
├── app.py                    # Streamlit web app
├── requirements.txt
├── src/
│   ├── train.py              # Data loading, model comparison, training, evaluation
│   └── recommend.py          # Looks up description/precautions/medications/diet
├── data/
│   ├── Training.csv          # 4,920 x 133 (132 symptoms + prognosis)

│   ├── precautions_df.csv

```

## Run locally
```bash
pip install -r requirements.txt
python src/train.py     # optional — retrains and overwrites model/ (~10-30s)
streamlit run app.py
```

## Deploy for free so anyone can access it

### Option A — Streamlit Community Cloud (recommended, easiest)
1. Create a free GitHub account if needed: https://github.com
2. Create a new **public** repo, e.g. `disease-prediction-pro`, and upload every file/
   folder in this project (drag-and-drop on github.com, or `git push`).
3. Go to https://share.streamlit.io → sign in with GitHub.
4. Click **"New app"** → pick your repo → branch `main` → main file path `app.py`.
5. Click **Deploy**. In ~1-2 minutes you get a public link:
   `https://your-app-name.streamlit.app`
6. Share it — anyone can open it, no login required.

### Option B — Hugging Face Spaces
1. Free account at https://huggingface.co
2. **New Space** → SDK: **Streamlit** → name it → visibility: **Public**.
3. Upload all files (or link a GitHub repo).
4. Space auto-builds and gives you:
   `https://huggingface.co/spaces/your-username/disease-prediction-pro`

Both are free, require no server maintenance, and auto-redeploy on every push.

## Where to take this further
- Swap in a real-world, noisier clinical dataset (e.g. MIMIC-derived or a hospital EHR
  export) to get a model whose accuracy actually reflects diagnostic difficulty.
- Add symptom severity weighting (already included as `Symptom-severity.csv`) into the
  feature vector instead of plain binary presence/absence.
- Add SHAP or permutation feature importance to explain *why* a prediction was made.
- Add input validation / a minimum-symptom-count threshold before showing a
  high-confidence prediction.
- Track real usage feedback (was the prediction useful?) to guide future retraining.
